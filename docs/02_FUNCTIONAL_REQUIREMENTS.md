# Functional Requirements Specification (FRS)

## UNICC AI Safety Evaluation Lab — Spring 2026

---

## 1. Purpose

This FRS defines the functional requirements for the AI Safety Lab,
a multi-module inference ensemble that evaluates AI agents for trust,
safety, and governance compliance within the UNICC AI Sandbox.

---

## 2. Functional Requirements

### FR-01: GitHub Repository Ingestion
- The system SHALL accept GitHub repository URLs as input
- The system SHALL clone repositories into isolated local directories
- The system SHALL detect project language, framework, and entry points
- The system SHALL classify project type (chatbot, agent, RAG, content_gen, etc.)

### FR-02: Sandboxed Target Model Execution
- The system SHALL execute target models in restricted environments
- Target models SHALL NOT have network access during execution
- Target models SHALL be subject to memory and time limits
- The system SHALL capture stdout, stderr, and generated files

### FR-03: Adversarial Test Generation
- The system SHALL provide a library of adversarial test prompts
- Prompts SHALL cover: prompt injection, harmful content, PII leakage,
  hate speech, governance violations, and safe baselines
- The system SHALL select appropriate prompts based on target model type

### FR-04: Llama Guard 3 Text Evaluation
- The system SHALL classify model outputs using Llama Guard 3 (8B)
- Classification SHALL use the MLCommons S1-S14 hazard taxonomy
- The system SHALL retry on transient failures (timeout, cold-start)
- Each judge SHALL evaluate only its assigned subset of categories

### FR-05: Llama Guard 3 Vision Evaluation
- The system SHALL evaluate images generated by target models
- Vision evaluation SHALL detect: obscene content, sensitive documents,
  PII in images, hateful imagery, violent imagery
- Vision evaluation SHALL use Llama Guard 3 Vision (11B) via Ollama

### FR-06: Security Assessment (Expert Module 1)
- SecurityJudge SHALL evaluate: prompt injection success, harmful content,
  PII leakage, code execution risks, data exfiltration, deception
- SecurityJudge SHALL use Llama Guard S1, S2, S7, S9, S14
- SecurityJudge SHALL supplement with PII regex scanning

### FR-07: Ethics Assessment (Expert Module 2)
- EthicsJudge SHALL evaluate: hate speech, discrimination, xenophobia,
  incitement, sexual content, self-harm, fairness
- EthicsJudge SHALL give special weight to UN humanitarian context
  (refugee/migrant protection, cultural sensitivity)
- EthicsJudge SHALL use Llama Guard S3, S4, S6, S10, S11, S12

### FR-08: Governance Assessment (Expert Module 3)
- GovernanceJudge SHALL evaluate: defamation, IP, political interference,
  application sensitivity, capability risk, deployment readiness
- GovernanceJudge SHALL incorporate results from other judges to assess
  holistic deployment readiness
- GovernanceJudge SHALL determine human oversight requirements

### FR-09: Council Orchestration
- The orchestrator SHALL coordinate all three judges independently
- The orchestrator SHALL apply conservative (strictest-wins) resolution
- The orchestrator SHALL detect and surface inter-judge disagreements
- Disagreements SHALL trigger mandatory human oversight flags

### FR-10: Safety Reporting
- The system SHALL produce JSON and human-readable text reports
- Reports SHALL include: summary, risk assessment, judge details,
  vision assessment, disagreements, recommendations
- The system SHALL export batch results as CSV
- The system SHALL produce requirements compliance matrices

### FR-11: Web Interface
- The system SHALL provide a web-based UI for submitting evaluations
- The UI SHALL show real-time progress during evaluation
- The UI SHALL render Turnitin-style detailed reports
- The UI SHALL display score breakdowns, risk levels, and recommendations

### FR-12: Health Monitoring
- The system SHALL provide a health check utility
- Health check SHALL verify: Ollama status, model availability,
  inference capability, dependencies, disk space
- Health check SHALL provide actionable fix instructions on failure

---

## 3. Non-Functional Requirements

### NFR-01: Standalone Operation
- The system SHALL operate without external API calls
- All inference SHALL run on-premises via Ollama on the DGX Spark cluster

### NFR-02: Auditability
- All evaluations SHALL be logged with full provenance
- Judge scores, rationale, and evidence SHALL be traceable

### NFR-03: Resilience
- SLM calls SHALL retry on transient failures
- The system SHALL gracefully degrade if a model is unavailable

### NFR-04: Performance
- Single evaluation SHALL complete within 5 minutes
- Full test suite (25+ prompts) SHALL complete within 60 minutes

### NFR-05: Security
- Target model execution SHALL be sandboxed
- No credentials SHALL be passed to target model environments
